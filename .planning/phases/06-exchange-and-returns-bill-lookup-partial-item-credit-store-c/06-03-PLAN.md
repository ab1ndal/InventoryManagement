## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| location.state (set by ExchangePage in same session) → BillingPage | Same-origin route state; not reachable by external attackers |
| BillingForm exchangeCredit prop → Summary | Parent-to-child prop; trusted pathway |
| BillingForm finalize → bills.notes | Admin-composed notes appended with exchange context |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-20 | T (Tampering) | location.state exchangeCredit.amount | mitigate | amount comes from ExchangePage which computed it from DB bill_items row via pure `calcItemCredit`. A malicious user with devtools could set a higher amount via `history.pushState`, but the credit was ALREADY written to `customers.store_credit` by ExchangePage before navigate — the amount here is only display/deduction on a new bill. It does NOT increase the customer's store_credit balance (that happened earlier). Worst case: staff finalizes a new bill with a fake deduction; this is caught in UAT (staff see unexpected low total) and is bounded by admin trust (RequireAdminAuth). |
| T-06-21 | I (Info disclosure) | window.history.replaceState | mitigate | replaceState({}, "") removes the state immediately after consumption — exchange credit does not persist into browser history. No PII leaked via bookmarks/shares. |
| T-06-22 | R (Repudiation) | exchange not traced on finalized bill | mitigate | `exchangeNote` appended to `bills.notes` on finalize captures the exchange reference (label + amount). Combined with `exchanges` table rows (Plan 02) there is a complete audit trail. |
| T-06-23 | D (DoS via back-nav loop) | route state persistence | mitigate | Pitfall 4 addressed — `window.history.replaceState({}, "")` prevents re-open on back navigation or page refresh. |
| T-06-24 | E (Elevation via abandoned exchange) | D-18 abandon flow | accept | Per D-18, if staff close BillingForm without finalizing, the customer's store_credit already holds the amount (written by ExchangePage).
