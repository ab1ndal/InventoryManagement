# Codebase Audit — 2026-07-07

Full audit of `retail-inventory` (React 19 SPA + Supabase, ~27k source lines). Findings verified against the **live database** (`Inventory-Management`, Mumbai) via anon/service-role probes, `supabase db advisors --linked`, a `pg_policies` dump, and full-source sweeps — not source reading alone.

**Severity counts:** 2 Critical · 8 High · 18 Medium · 10 Low · 6 Strengths

> **TL;DR:** one live data leak (cost prices publicly readable via `mockups_view`), one silent-UX bug (billing-page toasts never render), one 1,125-line dead file carrying a divergent GST engine, and a service-role key one accidental reference away from shipping in the browser bundle. RLS posture on core tables is otherwise solid, and the storefront UI is the best-built part of the app.
>
> **Round 2 (deeper reviews, §7–§12):** the billing finalize flow is a chain of ~10 un-transacted client-side writes with a textbook lost-update race on stock and TOCTOU on vouchers/credit — the highest-risk finding in the whole audit. The SMS/WhatsApp edge function has **no caller authorization** (any anon-key holder can send WhatsApp messages to arbitrary numbers on your Meta account). Billing math itself is correct and well-tested. Live data is clean (1 stale bill). Schema has real normalization debt: a `salesperson_ids` array duplicating a junction table, already **24 rows out of sync**.

## 1 · Security (verified live)

### 🔴 CRITICAL — Cost prices leak to the public via `mockups_view`

Verified live: an unauthenticated request with the anon key returns rows including **`purchaseprice`** — e.g. `{"productid":"BC25001","purchaseprice":1071.00}`. The view is `SECURITY DEFINER`, so it evaluates with its creator's privileges and bypasses RLS on the underlying `products`/`mockups` tables. Anyone (competitor, customer) can dump the entire cost sheet.

**Fix:** `ALTER VIEW public.mockups_view SET (security_invoker = true);` (or revoke anon/authenticated SELECT). If the storefront needs part of it, expose a narrow view without `purchaseprice`. This is the Supabase advisor's only ERROR-level finding.

### 🟠 HIGH — Service-role key carries the `REACT_APP_` prefix

`.env` defines `REACT_APP_SUPABASE_SERVICE_ROLE_KEY` (used by `scripts/migrate.js:26`). CRA inlines any *referenced* `REACT_APP_*` var into the public JS bundle at build time. Today nothing in `src/` references it — but one accidental import ships god-mode DB access to every browser, silently.

**Fix:** rename to `SUPABASE_SERVICE_ROLE_KEY` (no prefix) so CRA can never inline it; update `scripts/migrate.js` and the CLAUDE.md schema-fetch snippet.

### 🟡 MEDIUM — Supabase advisor warnings (12 distinct)

- **5 SECURITY DEFINER functions callable by anon/authenticated:** `is_admin`, `handle_new_user`, `set_bill_number`, `get_max_manual_item_suffix`, `sync_bill_salesperson_ids`. `is_admin` is fine by design; audit the rest — `set_bill_number` especially.
- **14 functions with mutable `search_path`** (`get_max_product_suffix`, `generate_bill_number`, …) — add `SET search_path = public`.
- **`prompts`** has an always-true RLS policy.
- **`mockups` storage bucket** allows public listing.
- **Auth config:** OTP expiry > 1 hour; leaked-password protection (HaveIBeenPwned) disabled.
- **Postgres `17.4.1.043`** has pending security patches — upgrade from dashboard.
- **Perf:** 5 policies re-evaluate `auth.uid()` per row (wrap as `(select auth.uid())`); multiple permissive policies on `products`/`categories`/`productsizecolors`/`profiles`; duplicate index on `productsizecolors`.

### 🟡 MEDIUM — RLS policies exist only in the live DB, not in the repo

The live database has a full `admin_only USING (is_admin())` policy set across bills, customers, suppliers, etc. — but `schema/` only commits storefront-read and storage policies. A rebuilt environment would come up with RLS enabled and *no* admin policies (total lockout), and there is no reviewable history of policy changes.

**Fix:** dump current policies into `schema/migration_rls_policies.sql` so repo matches production.

### ✅ STRENGTH — Core RLS posture is solid

Verified live: RLS enabled on every public table; anon reads return 0 rows and anon inserts get `42501` on customers/bills/suppliers/profiles/vouchers/etc. Storefront reads (`products`, `productsizecolors`, `categories`, `sizes`) work through explicit anon policies. Admin access gates on `is_admin()` (checks `profiles.role` + `is_active`), not bare `TO authenticated` — so the public `/admin/signup` page doesn't grant data access.

## 3 · Complex structures

### 🟠 HIGH — `BillingForm.js`: 2,492 lines, 44 `useState`, 12 `useEffect`

One component owns pricing, stock decrement, discount/voucher application, exchange credits, split payments, PDF generation, WhatsApp send, and QZ label printing. 44 independent state atoms means invalid intermediate states are always representable, and every keystroke re-renders the whole tree.

**Fix:** extract by responsibility (item editing / payment / finalize side-effects), and move the interdependent bill state into a single `useReducer` — the codebase already proves the pattern in `use-toast.ts`.

### 🟡 MEDIUM — Three more god components

- `BillTable.js` — 1,085 lines, 14 states: filters, pagination, PDF regen, WhatsApp/SMS send, delete flow.
- `SupplierTransactionDialog.js` — 1,034 lines: transaction CRUD + line items + bill-image upload in one dialog.
- `ExchangePage.js` — 792 lines, 17 states.

Meanwhile 243 raw `await supabase…` calls live inside components. The no-API-layer choice is documented, but at this scale each table's query logic is scattered across 5–9 files (e.g. `salespersons` queried from 9 places).

### 🟢 LOW — `mockups` table is a state machine encoded as 8 booleans

`redo, base_mockup, file_mockup, mockup, ig_reel, ig_post, video, whatsapp` — pipeline stages as independent flags allow contradictory combinations (e.g. posted-to-IG but no base mockup).

## 4 · Frontend — UI/UX

### 🟠 HIGH — Billing-page toasts never render; errors silently vanish

Two toast systems coexist: **sonner** (24 files, mounted globally in `App.js:80`) and the shadcn **`use-toast`** hook (used in `BillTable.js`, `ProductTable.js`). The shadcn `<Toaster/>` is mounted *only* on `InventoryPage.js:177`. **Fix:** migrate `BillTable`/`ProductTable` to sonner's `toast()`; delete `components/hooks/use-toast.ts`, `ui/toast.tsx`, `ui/toaster.tsx`.

### 🟡 MEDIUM — Errors swallowed to the console across the admin app

46 `if (error)` branches log to console only; 9+ call sites destructure `const { data } = await supabase…` and never look at the error (e.g. `BillingForm.js:507`, `ExchangePage.js:152`, `HistoryPage.jsx:37`). A failed fetch reads as "empty list" to the user.

**Fix:** minimum bar: `toast.error()` on every fetch/mutation failure; never drop the error from the destructure.

### 🟡 MEDIUM — Destructive actions use `window.confirm`

Bill delete (`BillTable.js:613`), customer delete (`CustomerTable.js:101`), supplier-transaction delete (`SupplierTransactionsTab.js:66`), bill-document delete (`SupplierTransactionDialog.js:147`) all use the native browser confirm — unstyled, blocking, inconsistent with the Radix dialogs used everywhere else, and no red destructive emphasis. **Fix:** `npx shadcn@latest add alert-dialog`, use it for all four.

### 🟡 MEDIUM — Global toast duration is 2 seconds

`App.js:80` sets `duration={2000}` for every toast, including errors with instructions in them. Guideline is 3–5s; error toasts should persist longer or require dismissal.

**Fix:** default 4000ms; pass longer/sticky duration for `toast.error`.

### 🟡 MEDIUM — Table UX inconsistent: pagination only in BillTable

`BillTable` paginates server-side; `ProductTable` and `CustomerTable` batch-fetch *all* rows into memory and render one giant list. Works today, degrades linearly with catalog/customer growth, and the two halves of the admin feel different.

### 🟢 LOW — Accessibility + input details

## 7 · Concurrency & atomicity (billing flows)

Traced the finalize-bill path in `BillingForm.js`. Because there is no server, a single "Finalize" click issues roughly **10 independent client-side writes in sequence** — no transaction, no rollback. Any crash, network drop, or RLS rejection mid-chain leaves a partially-written bill. The write sequence: insert `bills` → insert `bill_items` → (on item-insert failure) `delete bills` (manual compensation) → insert `bill_salespersons` → update `productsizecolors.stock` per variant → insert `discount_usage` → update `vouchers.redeemed` → update `customers.store_credit` → update `bills.pdf_url`.

### 🔴 CRITICAL — Stock decrement is a read-modify-write race (lost update)

`BillingForm.js:860–864` and `:982–983` do:
```js
const currentStock = stockMap[it.variantid]?.stock ?? 0;   // read (from state, fetched earlier)
await supabase.from("productsizecolors")
  .update({ stock: currentStock - it.quantity })            // write (absolute value)
  .eq("variantid", it.variantid);
```
The new stock is computed in JS from a value read earlier, then written as an **absolute number**. Two terminals (or two tabs) finalizing concurrently both read stock=5, both write 5−1=4 — one sale silently vanishes and inventory overstates by one. The check found 0 negative-stock rows today, but this is luck/low-volume, not safety. This is the single highest-risk defect in the audit: it corrupts inventory *and* under-reports sales.

**Fix:** move the decrement into an atomic SQL expression — an RPC doing `update productsizecolors set stock = stock - $qty where variantid = $v and stock >= $qty returning stock` (the `>= $qty` guard also prevents overselling and surfaces "insufficient stock" as a real error). Better: wrap the entire finalize in one `SECURITY DEFINER` RPC that takes the bill payload and does all inserts + decrements + redemptions in a single transaction, so partial writes become impossible.

### 🟠 HIGH — Voucher & store-credit redemption are TOCTOU-racy and non-idempotent

- **Voucher:** validity ("is it redeemed?") is checked at apply-time (`:641`) but the redeem-write happens much later at finalize (`:1409`). Between the two, the same voucher can be applied to a second bill — both finalize, both mark it redeemed, value spent twice. No `where redeemed = false` guard on the update.
- **Store credit:** `:1379` reads `store_credit`, subtracts in JS, writes absolute with `Math.max(0, …)`. Same lost-update race as stock; the `max(0,…)` clamp silently swallows over-spend instead of erroring.
- **Non-idempotent retries:** if finalize throws *after* stock/credit writes but before completion, a re-click runs the whole chain again — double-decrementing stock and credit. There is manual compensation for one failure mode only (`delete bills` when `bill_items` insert fails, `:960`, `:1217`); every other step is unguarded.

**Fix:** guard every state-transition write with a `where` predicate on the pre-state (`redeemed = false`, `stock >= qty`), and fold the whole flow into one transactional RPC so a retry is all-or-nothing.

### 🟢 LOW — ID generators are actually safe (verified)

Both bill-number paths are atomic at the DB: the `set_bill_number` trigger uses `INSERT … ON CONFLICT DO UPDATE SET last_value = last_value + 1 RETURNING` (single statement, no race). The older `generate_bill_number` uses `UPDATE … RETURNING`, also atomic. `get_max_product_suffix` (`select max(suffix)+1`) *is* racy under truly concurrent product creation, but product creation is rare/single-user — low priority. No duplicate bill numbers found live.

## 8 · Billing-math correctness

**Verdict: correct and genuinely well-tested — no action needed.** Reviewed `billUtils.js` against Indian textile GST rules and confirmed the test suite encodes the *rules*, not just current output.

- Unstitched → flat 5%; stitched → 5% at/below ₹2,500 per unit, 18% above — evaluated on the **post-discount** effective per-unit price (`:120–129`), which is the correct base. Tests `billUtils.test.js:32,53` assert both slabs with worked numbers (3000→18%→545 GST; 900→5%→45 GST).
- Alteration is treated as 5%-GST-inclusive and unbundled to pre-tax + 5% GST (`normalizeItem` `alterPreTax = gross/1.05`) — correct.
- Overall/voucher/balance discounts apply **pre-tax**, GST recomputed on the reduced taxable, with per-item proportional allocation (`:104–133`) — correct ordering (discount before tax).
- `computePreTaxBalanceDiscount` uses binary search to hit a target grand-total across the 18%→5% slab transition (`:73–84`) — a thoughtful solution to a genuinely non-linear problem.
- `InvoiceView.js:104` splits GST into CGST/SGST halves for the printed invoice — correct for intra-state Karnataka sales.

The only latent risk is the two-engine divergence already filed in §2 (the dead `archive/BillForm.js` defaults GST to 12%). Delete it so it can never be the one that runs.

## 10 · Edge functions & storage ACL

### 🔴 CRITICAL — `send-bill-sms` edge function has no caller authorization

`supabase/functions/send-bill-sms/index.ts` accepts a POST with `{ phone, customerName, billNumber, amount, pdfUrl }` and immediately calls the Meta WhatsApp Graph API with your server-side `WHATSAPP_ACCESS_TOKEN`. It **never verifies the caller** — no JWT check, no `is_admin()`, and `Access-Control-Allow-Origin: *`. Supabase deploys functions with JWT verification on by default, but nothing *inside* the function enforces role, so any holder of the public anon key (which ships in the browser bundle) can:

- Send WhatsApp template messages to **arbitrary phone numbers** on your business account — spam/abuse that burns your Meta messaging quota and risks account suspension.
- Supply an arbitrary `pdfUrl` parameter that gets injected into the message body — phishing vector (messages appear to come from Bindals Creation).

**Fix:** verify the caller inside the function — extract the JWT from the `Authorization` header, confirm the user's `profiles.role` is admin/superadmin before sending. Lock CORS to the admin origin. Ideally look up the phone number and bill server-side from `billNumber` rather than trusting client-supplied `phone`/`amount`/`pdfUrl`.

### 🟡 MEDIUM — `mockups` storage bucket is public and listable

Confirmed live: bucket `mockups` is `public=true` and anon can enumerate every folder (`BC25001`, `BC25002`, …). Product mockup images are arguably meant to be public (storefront uses them), but **directory listing** exposes your full catalog structure and unreleased product IDs. `invoices` (`public=false`) and `supplier-bills` (`public=false`) correctly reject anon listing — verified `invoices` returns `[]` under the anon key. 

**Fix:** if images must stay public, disable listing (serve via known paths / signed URLs) so the catalog can't be enumerated.

### 🟢 LOW — No `supabase/config.toml` in repo

The functions directory exists but there's no committed `config.toml`, so function JWT-verification settings and local dev config aren't reproducible. Add one.

## 12 · Database normalization

Schema is largely sound (3NF for the core transactional tables), but carries three denormalization decisions that have already caused live drift.

### 🟠 HIGH — `bills.salesperson_ids` array duplicates the `bill_salespersons` junction — 24 rows already out of sync

Salesperson attribution is stored **twice**: normalized in the `bill_salespersons` junction table, and denormalized as an `int4[]` array column `bills.salesperson_ids` kept in sync by the `sync_bill_salesperson_ids` trigger (which rebuilds the array from `bill_items.salesperson_id`, a *third* location). Live check: **24 bills where the array length disagrees with the junction row count.** The redundancy is the classic normalization anti-pattern — the same fact in three places with no single source of truth, and they already diverge.

**Fix:** pick one source of truth. If array reads are needed for convenience, make the junction authoritative and treat the array as a derived cache (and fix the trigger to rebuild from the junction, not from `bill_items`). Reconcile the 24 drifted rows.

### 🟡 MEDIUM — Weak domain constraints on `bills`

Zero CHECK constraints on `bills`. `paymentstatus` is a free-text `varchar` (only 1 distinct value live, so no corruption yet, but nothing enforces the `draft/partial/finalized/cancelled` state machine the code assumes). Money columns (`totalamount`, `net_amount`, `final_amount`, `store_credit_used`, …) have no non-negative checks. Combined with §7's non-atomic writes, there's no DB-level backstop against the exact inconsistency found in §9.

**Fix:** convert `paymentstatus` to an enum (or CHECK-constrained), add `>= 0` checks on money columns, add a deferred check or trigger that `totalamount = Σ bill_items.total` for finalized bills.

### 🟢 LOW — `discounts.category` / snapshot columns

- `discounts.category` and `discounts.product_ids _text` embed catalog references as free text / text-array rather than FKs to `categories`/`products` — a code deleted or renamed here leaves dangling references. Acceptable for a small discount table, but worth a periodic validation query.
- `bill_items` snapshots `product_name`, `product_code`, `category`, `mrp`, `cost_price` at sale time. This is **correct** denormalization (an invoice must freeze historical values even if the product later changes) — noted here only to distinguish it from the `salesperson_ids` case, which is *not* a legitimate snapshot.
- `mockups` 8-boolean pipeline (from §3) is also a normalization smell — those stages belong in a status enum or a per-stage event table.

### ✅ STRENGTH — Core transactional model is clean

`bills → bill_items → productsizecolors` is properly normalized with real FKs (verified: 24 tables carry FK constraints). Sale-time snapshots on `bill_items` are done right. Junction tables exist where they should (`bill_salespersons`, `discount_usage`). The problems are additive denormalizations on top of a sound core, not a broken core.

## Phase 1 — Billing correctness & atomicity (the core risk, §7 + §9)

This is the highest-value work in the audit. Two sub-steps: a fast atomic-stock stopgap, then the full transactional finalize.

### 1.1 — Atomic stock decrement RPC (stopgap) · **M**
Replace the read-modify-write in `BillingForm.js` with a DB-atomic conditional decrement.
```sql
-- schema/migration_apply_stock_delta_rpc.sql
CREATE OR REPLACE FUNCTION public.apply_stock_delta(p_variantid uuid, p_delta numeric)
RETURNS numeric
LANGUAGE plpgsql SECURITY DEFINER SET search_path = public
AS $$
DECLARE new_stock numeric;
BEGIN
  IF NOT is_admin() THEN RAISE EXCEPTION 'not authorized'; END IF;
  UPDATE productsizecolors
    SET stock = stock + p_delta               -- delta is negative for a sale
    WHERE variantid = p_variantid
      AND stock + p_delta >= 0                 -- guard: blocks overselling
    RETURNING stock INTO new_stock;
  IF new_stock IS NULL THEN
    RAISE EXCEPTION 'insufficient stock for variant %', p_variantid USING ERRCODE = 'check_violation';
  END IF;
  RETURN new_stock;
END; $$;
REVOKE EXECUTE ON FUNCTION public.apply_stock_delta(uuid, numeric) FROM PUBLIC, anon;
GRANT EXECUTE ON FUNCTION public.apply_stock_delta(uuid, numeric) TO authenticated;
```
In `BillingForm.js` (`:860`, `:982`, `:1240`, `:1287`) swap the `update({ stock: currentStock - qty })` calls for `supabase.rpc("apply_stock_delta", { p_variantid, p_delta })`, surface the `insufficient stock` error as a toast, and abort finalize on it. **Verify:** unit-simulate two concurrent decrements from stock=1 → exactly one succeeds, the other errors. Confirms the lost-update and oversell paths are both closed.

### 1.2 — Transactional finalize RPC (the real fix) · **L**
Fold the ~10 client writes into one server-side transaction so a bill is all-or-nothing. Signature roughly:
```
finalize_bill(p_bill jsonb, p_items jsonb, p_salespersons int[],
              p_codes text[], p_voucher_id text, p_store_credit_used numeric)
  RETURNS bills   -- SECURITY DEFINER, is_admin() gate
```
Inside one function body (implicitly one transaction): insert/update `bills`; insert `bill_items`; call `apply_stock_delta` per line (its guard now protects the whole flow); guarded voucher redeem `UPDATE vouchers SET redeemed=true … WHERE voucher_id=$1 AND redeemed=false` (0 rows updated ⇒ `RAISE` and roll back); guarded credit debit `UPDATE customers SET store_credit = store_credit - $used WHERE customerid=$c AND store_credit >= $used`; insert `discount_usage`. Any `RAISE` rolls back every prior write — the partial-bill class (the `FY26-000012` corruption) becomes impossible. Client `handleFinalize` shrinks to build-payload → `rpc("finalize_bill", …)` → handle result, and the manual `delete bills` compensation code is deleted. **Verify:** finalize a bill normally (works); force a mid-way failure (bad voucher) and confirm **no** `bills` row, **no** stock movement persisted; re-run the §9 integrity SQL → still 0 mismatches; existing `billUtils.test.js` still green.
